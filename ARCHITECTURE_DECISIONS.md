# Архітектурні рішення SCLOC-Verse (ADR)

Цей документ фіксує ключові архітектурні рішення проєкту: не лише **що** зроблено,
а **чому** обрано саме так. Оновлюється при прийнятті нових архітектурних рішень.

Формат запису: **Контекст → Рішення → Наслідки**.

> Документ створено за результатами архітектурного аудиту (2026-06). Раціонал
> спирається на спостереження у коді; автор проєкту може доповнювати деталі.

---

## ADR-001. Ручний Composition Root замість IoC-контейнера

**Контекст.** Десктопний WPF-додаток із чітко окресленим набором сервісів.
Критично важливий прозорий життєвий цикл і порядок `Dispose` (фонові таймери,
HttpListener, Supabase refresh, overlay-вікно).

**Рішення.** `AppCompositionRoot` + `AuthCompositionRoot` із ручним `new`-компонуванням
залежностей. Без сторонніх IoC-контейнерів.

**Наслідки.**
- ➕ Повна прозорість, явний dispose-каскад, відсутність «магії».
- ➖ При зростанні кількості сервісів конструктори споживачів ростуть
  (див. ADR-004 про MainWindow).

---

## ADR-002. Два бекенди гарячих клавіш (RegisterHotKey + Raw Input)

**Контекст.** Глобальні гарячі клавіші мають спрацьовувати незалежно від фокусу вікна.
WinAPI пропонує два механізми з різними можливостями й обмеженнями.

**Рішення.** Абстракція `IHotkeyBackend` + дві реалізації
(`RegisterHotkeyBackend`, `RawInputBackend`) + `IHotkeyMessageSource` для ізоляції
бекендів від UI-технології.

**Наслідки.**
- ➕ Можливість вибору бекенду конфігурацією/змінною середовища
  (`SCLOCVERSE_HOTKEY_BACKEND`), A/B-порівняння, наявність запасного механізму.
- ➖ Два кодові шляхи доводиться підтримувати паралельно.

---

## ADR-003. Raw Input — бекенд за замовчуванням

**Контекст.** `RegisterHotKey` має суттєві обмеження: реєстрація конкретних жестів
(конфлікти з іншими застосунками за ту саму комбінацію), відсутність повідомлення
про відпускання клавіші. `RawInput` через `RIDEV_INPUTSINK` отримує `WM_INPUT`
незалежно від фокусу й підтримує key-up.

**Рішення.** `RawInputBackend` — бекенд за замовчуванням; `RegisterHotkeyBackend` —
запасний, вмикається конфігурацією.

**Наслідки.**
- ➕ Глобальне виявлення без попередньої реєстрації жестів; підтримка key-up,
  необхідна для `SuppressAutoRepeat` (див. ADR-008).
- ➖ Бекенд споживає всі `WM_INPUT`; всередині фільтруються лише відомі клавіші,
  щоб не обробляти/не журналювати довільне введення.

## ADR-004. MainWindow як UI-координатор (WPF code-behind)

**Контекст.** Головне вікно — природна точка зʼєднання UI і сервісів у WPF.

**Рішення.** `MainWindow` залишається code-behind-координатором Canvas-ів; бізнес-логіка
живе в сервісах і presenter-ах (`UpdateStatusPresenter`, `AuthStatusPresenter` тощо).

**Наслідки.**
- ➕ Простота й читабельність, прийнятна для WPF-конвенції.
- ➖ Фактично єдиний «важкий» файл; розширення має відбуватися через
  presenters/Canvas-и, а не роздування code-behind.

---

## ADR-005. Overlay як окреме вікно

**Контекст.** Таймер має відображатися поверх гри з click-through, перетягуванням,
масштабом і прозорістю.

**Рішення.** `HangarOverlayWindow` (окреме topmost-вікно) + `HangarOverlayService`;
Win32-стилі `WS_EX_TRANSPARENT`/`WS_EX_LAYERED` для click-through.

**Наслідки.**
- ➕ Незалежність від фокусу, повний контроль Win32-стилів вікна.
- ➖ Необхідно слідкувати за підписками подій (див. фікс витоку `PropertyChanged`
  у Phase 1 аудиту).

---

## ADR-006. Canvas-підхід навігації

**Контекст.** Навігація між секціями без мультівіконності.

**Рішення.** `CanvasManager` перемикає видимість Canvas-ів у межах одного `MainWindow`.

**Наслідки.**
- ➕ Миттєва навігація, єдиний стан вікна, простий layout.
- ➖ Усе UI-дерево живе в одному вікні; керується через окремі Canvas-контроли.

---

## ADR-007. Без Microsoft.Extensions.Hosting (Generic Host)

**Контекст.** Десктопний інструмент; lifetime керується `WPF Application` +
`AppCompositionRoot`; фонові задачі — через `DispatcherTimer`/`Task`.

**Рішення.** Не використовувати Generic Host.

**Наслідки.**
- ➕ Менше шаблонного коду, повний ручний контроль dispose.
- ➖ Якщо зʼявляться довготривалі фонові сервіси з потребою в уніфікованих
  конфігурації/логуванні/DI — рішення варто переглянути.

---

## ADR-008. Виділення `IKeyStateBackend` (Interface Segregation)

**Контекст.** `IHotkeyBackend` вимагав подію `KeyUp`, але `RegisterHotKey` її
фізично не підтримує (WinAPI віддає лише `WM_HOTKEY`) — це породжувало
попередження `CS0067` (подія заявлена, але ніколи не викликається). Водночас
`KeyUp` є навантаженим: сервіс використовує його для скидання стану
`SuppressAutoRepeat`.

**Рішення.** Винести `KeyUp` в окремий інтерфейс `IKeyStateBackend`. Його реалізує
лише `RawInputBackend`; `RegisterHotkeyBackend` — ні. `HotkeyService` підписується
умовно: `if (_backend is IKeyStateBackend keyState) keyState.KeyUp += …`.

**Наслідки.**
- ➕ Чесний контракт; `CS0067` усунуто; `SuppressAutoRepeat` коректно працює для
  Raw Input.
- ➖ Новий бекенд із підтримкою key-up має явно реалізувати `IKeyStateBackend`.

# Технічний борг

Записи відкладених змін із зазначенням причини відкладення та умови виконання.
Поточний принцип — **Zero Regression**: стабільний код не чіпаємо без потреби.

## TECHDEBT-001. Прибрати `WindowsAPICodePack.Shell.CommonFileDialogs`

**Стан.** Відкладено за рішенням (Zero Regression).

**Опис.** Пакет не має прямих використань у коді, але **транзитивно** надає
посилання на `System.Windows.Forms`, яким реально користується
`MainWindow.BtnSelectFolder_Click` (`System.Windows.Forms.FolderBrowserDialog`).
Чисте видалення пакета ламає збірку; глобальне `<UseWindowsForms>` породжує ~78
конфліктів неоднозначності з WPF-типами (`Button`, `TextBox`, `Brush`, …).

**Плановане рішення.** Мігрувати єдине використання `FolderBrowserDialog` на
нативний WPF `Microsoft.Win32.OpenFolderDialog` (.NET 8+). Це дозволить прибрати
**і NuGet, і залежність WinForms**.

**Причина відкладення.** Поточна реалізація стабільна й не впливає на користувачів;
`OpenFolderDialog` має іншу поведінку (початкова директорія, відновлення останнього
шляху, UNC/мережеві папки) і потребує повного тестування.

**Коли виконувати.** Під час великого релізу (напр. SCLOC-Verse 2.0) або рефакторингу
модулів settings/installer, де зміна буде однією із запланованих і протестованих.

**Виявлено.** Phase 1 архітектурного аудиту.
