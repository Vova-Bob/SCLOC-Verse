# SCLocalizationUA Development Rules

Ти працюєш над проєктом SCLocalizationUA.

Перед початком роботи врахуй результати форензик аналізу архітектури, історії Git, стилю автора та структури проєкту.

## Основний принцип

Мета не переписати проєкт під "ідеальну архітектуру".

Мета — розвивати проєкт у стилі автора, зберігаючи стабільність, читабельність та сумісність існуючого коду.

Новий код повинен виглядати так, ніби його написав початковий автор проєкту.

---

# Обов'язковий порядок роботи

Перед будь-якими змінами виконувати:

1. Форензик аналіз задачі.
2. Аналіз ризиків.
3. План змін.
4. Очікування погодження.
5. Резервний commit.
6. Реалізація.
7. Короткий звіт.
8. Фінальний commit.

Заборонено писати код до погодження плану змін.

---

# Форензик аналіз задачі

Перед реалізацією визначити:

* які файли будуть змінені;
* які сервіси зачіпаються;
* які модулі можуть постраждати;
* ризики регресії;
* вплив на UI;
* вплив на архітектуру;
* вплив на сумісність.

Формат:

## Форензик аналіз

### Зона впливу

### Ризики

### Регресії

### Рекомендації

# Git правила

Перед змінами сформувати:

git add .
git commit -m "Резервна точка перед <опис зміни>"

Після завершення:

git add .
git commit -m "<короткий опис українською>"

Усі commit-повідомлення українською мовою.

---

# Архітектурні правила

Поточна архітектура є базовою та приймається без глобальних змін.

Заборонено:

* переписувати проєкт на іншу архітектуру;
* впроваджувати MVVM-фреймворки;
* впроваджувати сторонні IoC-контейнери;
* додавати складні патерни без необхідності;
* виконувати глобальний рефакторинг без погодження.

Використовувати існуючий підхід:

* AppCompositionRoot;
* ручна композиція залежностей;
* функціональне групування папок;
* Canvas-підхід;
* існуючий стиль XAML.

---

# Правила коду

Код:

* тільки англійською мовою;
* професійний C#;
* .NET 9;
* Nullable Enabled;
* DRY;
* KISS;
* SOLID без фанатизму.

Коментарі:

* тільки українською мовою.

Документація:

* українською мовою.

---

# Async правила

Async методи:

* обов'язково суфікс Async;
* CancellationToken додавати де це доцільно.

Сервіси:

.ConfigureAwait(false)

UI:

.ConfigureAwait(true)

Не використовувати async void окрім обробників подій WPF.

---

# Правила структури проєкту

Нові сервіси:

Services/<Feature>/

Нові інтерфейси:

Interfaces/

Нові моделі:

Models/

Нові Canvas:

Controls/

Нові залежності реєструвати через:

AppCompositionRoot

---

# Правила розвитку проєкту

Не збільшувати відповідальність MainWindow.xaml.cs без необхідності.

Нову логіку по можливості розміщувати:

* у сервісах;
* у відповідних Canvas;
* у допоміжних класах.

Не переносити існуючу логіку між шарами без погодження.

Не виконувати рефакторинг лише заради "кращих практик".

---

# Правила стабільного коду

Якщо файл:

* давно не змінювався;
* не має відкритих дефектів;
* працює стабільно,

вважати його стабільним.

Не змінювати без прямої необхідності.

---

# Правила виправлення помилок та неймінгу

Опечатки необхідно виправляти.

Перед виправленням визначити:

* чи використовується назва публічно;
* чи є зовнішні залежності;
* чи не виникне регресія.

Якщо ризик мінімальний:

* виправити.

Якщо ризик підвищений:

* провести форензик аналіз;
* показати ризики;
* отримати погодження.

Не залишати очевидні помилки типу:

* Resul → Result
* Cash → Cache
* Setupe → Setup
* Instal → Install

---

# Робота з чужими модулями

Перед зміною модулів інших розробників:

* виконати форензик аналіз;
* показати ризики;
* отримати погодження.

Особливо уважно ставитися до модулів:

* L.I.A
* Updater
* Assistant
