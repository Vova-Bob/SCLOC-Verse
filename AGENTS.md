# SCLOC-Verse Development Rules

Ти працюєш над проєктом SCLOC-Verse.

Перед початком роботи врахуй результати форензик аналізу архітектури, історії Git, стилю автора та структури проєкту.

## Основний принцип

Мета не переписати проєкт під "ідеальну архітектуру".

Мета — розвивати проєкт у стилі автора, зберігаючи стабільність, читабельність та сумісність існуючого коду.

Новий код повинен виглядати так, ніби його написав початковий автор проєкту.

---

# Обов'язковий порядок роботи

Перед будь-якими змінами виконувати:

1. Форензик аналіз задачі.
2. Аналіз ризиків.
3. План змін.
4. Очікування погодження.
5. Резервний commit.
6. Реалізація.
7. Короткий звіт.
8. Фінальний commit.

Заборонено писати код до погодження плану змін.

# Форензик аналіз задачі

Перед реалізацією визначити:

* які файли будуть змінені;
* які сервіси зачіпаються;
* які модулі можуть постраждати;
* ризики регресії;
* вплив на UI;
* вплив на архітектуру;
* вплив на сумісність;
* вплив на кодування тексту (особливо при роботі з українською мовою, GitHub API, JSON, XAML, .iss, .ps1).

Формат:

## Форензик аналіз

### Зона впливу

### Ризики

### Регресії

### Рекомендації

# Git правила

Перед змінами сформувати:

git add .
git commit -m "Резервна точка перед <опис зміни>"

Після завершення:

git add .
git commit -m "<короткий опис українською>"

Усі commit-повідомлення українською мовою.

# Правила кодування тексту (P0)

Це критичне правило. Пошкодження українського тексту (mojibake) не допускається ні в коді, ні в релізних артефактах, ні в GitHub API.

## Джерела ризику

* PowerShell за замовчуванням використовує кодування консолі (часто Windows-1251 або OEM), а не UTF-8.
* `Invoke-WebRequest -Body $string` кодує тіло запиту в кодуванні консолі, яке GitHub API не розуміє для українських символів.
* `Out-File`, `Set-Content` без `-Encoding UTF8` можуть записати файл у Windows-1251.
* `File.ReadAllText` без явного `Encoding.UTF8` на системі з не-UTF8 локаллю прочитає файл неправильно.
* Редактори, які зберігають XAML/JSON/C# файли без BOM або в ANSI, псують українські символи.

## Обов'язкові вимоги

1. **Всі C# / XAML / JSON / .iss / .ps1 файли** мають бути збережені у **UTF-8**.
2. **Усі українські тексти в коді** мають бути UTF-8 з коректними кодами символів, без mojibake (`Р›Р°СЃРєР°РІРѕ`, `??????` тощо).
3. **Читання текстових файлів** виконувати тільки з явним `Encoding.UTF8`:
   ```csharp
   File.ReadAllText(path, Encoding.UTF8)
   await File.ReadAllTextAsync(path, Encoding.UTF8)
   new StreamReader(path, Encoding.UTF8)
   ```
4. **Запис текстових файлів** виконувати тільки з явним `Encoding.UTF8`:
   ```csharp
   File.WriteAllText(path, content, Encoding.UTF8)
   await File.WriteAllTextAsync(path, content, Encoding.UTF8)
   ```
5. **PowerShell — запис файлів**:
   ```powershell
   $content | Set-Content -Path $file -Encoding UTF8
   [System.Text.Encoding]::UTF8.GetBytes($content) | Set-Content -Path $file -Encoding Byte
   ```
6. **PowerShell — веб-запити з JSON**:
   ```powershell
   $json = @{ body = "Український текст" } | ConvertTo-Json
   $bytes = [System.Text.Encoding]::UTF8.GetBytes($json)
   Invoke-WebRequest -Uri $url -Method POST -Body $bytes -ContentType "application/json; charset=utf-8" -Headers @{ ... }
   ```
   Ніколи не передавати український текст в `-Body $string` напряму.
7. **GitHub API release notes** створювати/оновлювати тільки з UTF-8 байтами та заголовком `Content-Type: application/json; charset=utf-8`.

## Перевірка перед commit / релізом

* Запускати `dotnet build` та візуально перевіряти всі екрани з українським текстом.
* Перевіряти GitHub release notes в браузері одразу після публікації.
* Якщо знайдено `Р›Р°СЃРєР°РІРѕ`, `Ð›Ð°`, `??????` — це P0 регресія, виправляти негайно.

## Наслідки порушення

* Пошкоджений український текст у продукті = P0 дефект.
* Пошкоджені release notes = непрофесійний публічний реліз.
* Mojibake в commit історії ускладнює подальшу роботу з кодом.

# Архітектурні правила

Поточна архітектура є базовою та приймається без глобальних змін.

Заборонено:

* переписувати проєкт на іншу архітектуру;
* впроваджувати MVVM-фреймворки;
* впроваджувати сторонні IoC-контейнери;
* додавати складні патерни без необхідності;
* виконувати глобальний рефакторинг без погодження.

Використовувати існуючий підхід:

* AppCompositionRoot;
* ручна композиція залежностей;
* функціональне групування папок;
* Canvas-підхід;
* існуючий стиль XAML.

---

# Правила коду

Код:

* тільки англійською мовою;
* професійний C#;
* .NET 9;
* Nullable Enabled;
* DRY;
* KISS;
* SOLID без фанатизму.

Коментарі:

* тільки українською мовою.

Документація:

* українською мовою.

---

# Async правила

Async методи:

* обов'язково суфікс Async;
* CancellationToken додавати де це доцільно.

У сервісах використовувати `ConfigureAwait(false)` там, де це дійсно потрібно.
Не додавати `ConfigureAwait` механічно.

У UI-шарі використовувати стандартний `await`, якщо немає спеціальної причини для іншої поведінки.

Не використовувати async void окрім обробників подій WPF.

# Правила структури проєкту

Нові сервіси:

Services/<Feature>/

Нові інтерфейси:

Interfaces/

Нові моделі:

Models/

Нові Canvas:

Controls/

Нові залежності реєструвати через:

AppCompositionRoot

---

# Правила розвитку проєкту

Не збільшувати відповідальність MainWindow.xaml.cs без необхідності.

Нову логіку по можливості розміщувати:

* у сервісах;
* у відповідних Canvas;
* у допоміжних класах.

Не переносити існуючу логіку між шарами без погодження.

Не виконувати рефакторинг лише заради "кращих практик".

---

# Правила розширення існуючого коду

Не створювати нові файли без необхідності.

Не створювати нові сервіси, інтерфейси або моделі, якщо існуючі вже вирішують задачу.

Перевага віддається розширенню існуючого коду.

Перед написанням коду завжди аналізувати існуючу реалізацію.
Якщо схожа логіка вже існує в проєкті — використовувати існуючий підхід автора.
Не вигадувати нову реалізацію, якщо аналог вже є.

---

# Правила стабільного коду

Якщо файл:

* давно не змінювався;
* не має відкритих дефектів;
* працює стабільно,

вважати його стабільним.

Не змінювати без прямої необхідності.

---

# Правила виправлення помилок та неймінгу

Опечатки необхідно виправляти.

Перед виправленням визначити:

* чи використовується назва публічно;
* чи є зовнішні залежності;
* чи не виникне регресія.

Якщо ризик мінімальний:

* виправити.

Якщо ризик підвищений:

* провести форензик аналіз;
* показати ризики;
* отримати погодження.

Не залишати очевидні помилки типу:

* Resul → Result
* Cash → Cache
* Setupe → Setup
* Instal → Install

---

# Робота з чужими модулями

* отримати погодження.

Особливо уважно ставитися до модулів:
